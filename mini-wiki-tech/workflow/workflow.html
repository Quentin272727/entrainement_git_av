Workflow Git simple

Ceci est une explication concise d'un workflow Git courant pour travailler en équipe : on développe sur des branches de fonctionnalité (feature branches), puis on ouvre une Pull Request (PR) pour revue, et enfin on fusionne (merge) dans la branche principale.

Résumé en 4 étapes

  1. Créer une branche de fonctionnalité depuis main (ou develop selon le projet).
  2. Travailler localement : commits atomiques et messages clairs, pousser la branche distante.
  3. Ouvrir une Pull Request (PR) pour revue et tests automatisés.
  4. Après approbation : fusion (merge) dans main, puis supprimer la branche de fonctionnalité si elle n'est plus utile.

Exemple de commandes

# se placer sur main
git checkout main
git pull origin main

# créer et basculer sur la branche de feature
git checkout -b feat/ma-fonctionnalite

# développer, ajouter, committer
git add .
git commit -m "feat: ajouter la fonctionnalité X"

# pousser la branche
git push -u origin feat/ma-fonctionnalite

# ouvrir une PR sur la plateforme (GitHub/GitLab/Bitbucket)

Bonnes pratiques

  • Commits petits et décrits clairement.
  • Relectures de code (code review) obligatoires pour les changements significatifs.
  • Tests automatiques (CI) avant fusion.
  • Supprimer les branches fusionnées pour garder le dépôt propre.

Flux simple : main ← merge ← feature/* → PR → review → merge
